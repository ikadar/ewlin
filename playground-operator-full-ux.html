Progress & Delay Signal Flow
Operators report progress from their station tablet. The system projects completion time and alerts when delays are likely. Operators signal the scheduler, who decides how to respond.
👨‍🔧 Operator (Station Tablet)
STEP 1
Log in & see queue
Opens /station/komori-g37 on mounted tablet. Sees today's tasks in order. Current task is highlighted.
↓
STEP 2
Report progress
Taps +500 each time a stack completes, or types exact counter value via keypad. Progress updates are timestamped and stored.
↓
STEP 3
Alert appears
System projects finish time from progress rate. If projected finish exceeds expected end by threshold → amber/red banner slides in.
↓
STEP 4
Signal delay
Operator picks a reason (slow run, machine issue, material) and taps "Signal Delay to Scheduler". Creates a DelaySignal record.
→
↓
STEP 5
Continue working
Operator keeps reporting progress. Banner shows "Signal sent" confirmation. When task is done, taps "Mark Complete" → next task auto-loads.

↓
Store progress log, compute %
↓
Client-side projection every 60s
↓
Persist DelaySignal & notify scheduler
↓
Update tile bars + badges on grid
📅 Scheduler (Desktop)
(Working on scheduling grid as usual)
↓
See live progress
Tiles on the grid show a thin progress bar at the bottom. Color shifts from green → amber → red as pace falls behind.
↓
Pulsing badge appears
Even without the signal, the scheduler can see tiles at risk via amber/red badges and progress bar colors.
↓
NOTIFICATION
Delay signal received
Badge count increments. Toast notification appears. Delay Signals panel shows details + operator's reason.
↓
RESPONSE
Scheduler decides
Extend (+15m, +30m, +1h) — updates task duration, triggers conflict detection for downstream tasks.
Acknowledge — dismiss without action.
Rearrange — use existing tools to move downstream tasks.
