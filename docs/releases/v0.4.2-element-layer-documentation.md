## Overview

This release updates all domain model and architecture documentation to reflect the Element layer introduced in v0.4.0–v0.4.1. The Element entity is an intermediate layer between Job and Task that enables multi-element jobs (e.g., book production with cover, interior, and binding elements), element-scoped task sequencing, and cross-element finish-to-start dependencies.

No code changes — documentation only.

## Technical Notes

### Key Concepts to Document

1. **Element entity:** Intermediate layer between Job and Task. Every job has at least one element. Fields: `id`, `jobId`, `suffix`, `label?`, `prerequisiteElementIds`, `taskIds`.

2. **Intra-element precedence:** Tasks within the same element follow `sequenceOrder` (linear sequence). This replaces the old job-wide flat task sequence.

3. **Cross-element precedence:** `prerequisiteElementIds` defines finish-to-start dependencies between elements in the same job. ALL tasks of a prerequisite element must complete before the dependent element's first task can start.

4. **Dry time:** If a prerequisite element's last task is a printing task, the dependent element's first task must wait an additional 4h drying time.

5. **Task.elementId replaces Task.jobId:** Tasks reference their Element via `elementId`; the Job is accessible via `Element.jobId`.

6. **Single-element backward compatibility:** Single-element jobs behave identically to the pre-Element flat task list. The element layer is transparent.

7. **`isMultiElementJob(elementIds)`:** Helper function returning `true` when `elementIds.length > 1`.

## Change Description

> **Source of truth** for all documentation updates in this release. Based on code analysis of v0.4.0–v0.4.1 changes.

### New Entity: Element (`packages/types/src/element.ts`)

The **Element** is an intermediate layer between Job and Task that enables multi-element jobs (e.g., book production with cover, interior, and binding elements).

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | Unique identifier |
| `jobId` | `string` | Parent Job ID |
| `suffix` | `string` | Short identifier (e.g., `"couv"`, `"int"`, `"fin"`, `"ELT"`) |
| `label?` | `string` | Display label (e.g., `"Couverture"`, `"Intérieur"`) |
| `prerequisiteElementIds` | `string[]` | Finish-to-start dependencies (same-job elements) |
| `taskIds` | `string[]` | Task IDs in execution order within element |
| `createdAt` | `string` | ISO timestamp |
| `updatedAt` | `string` | ISO timestamp |

Helper: `isMultiElementJob(elementIds: string[]): boolean` — returns `true` when `elementIds.length > 1`.

### Task.elementId Replaces Task.jobId (`packages/types/src/task.ts`)

- `BaseTask.jobId` **removed**, replaced by `elementId: string`
- Navigation path: `Task.elementId → Element.jobId → Job`
- `sequenceOrder: number` — position within element's task sequence (0-indexed)

### Job.elementIds Added (`packages/types/src/job.ts`)

- `Job.elementIds: string[]` — list of element IDs belonging to this job
- `Job.taskIds` deprecated — use `Element.taskIds` instead

### ScheduleSnapshot Extended (`packages/types/src/assignment.ts`)

- `ScheduleSnapshot.elements: Element[]` — all elements in the snapshot

### Intra-Element Precedence (Within Same Element)

- `sequenceOrder` determines task order within an element
- Strict linear sequence: Task N+1 cannot start until Task N completes
- Replaces old job-wide flat task sequence — now **element-scoped**

### Cross-Element Precedence (Between Elements)

Controlled by `prerequisiteElementIds` (`packages/validator/src/`):

- Only the element's **first task** can be blocked by prerequisite elements
- Only the element's **last task** can block dependent elements
- Rule: All tasks of prerequisite element must complete before dependent element's first task can start
- Helpers: `getCrossElementPredecessors()`, `getCrossElementSuccessors()` in `utils/helpers.ts`

### Dry Time with Elements

- `DRY_TIME_MINUTES = 240` (4 hours)
- If predecessor is a printing task (offset press category: `cat-offset`), successor waits +4 hours
- Physical process: continues outside working hours
- Applies to both intra-element and cross-element precedence

### Frontend Utilities (`apps/web/src/utils/taskHelpers.ts`)

- `getJobIdForTask(task, elements)` — Task → Job ID navigation
- `getTasksForJob(jobId, tasks, elements)` — all tasks for a Job
- `getElementTasks(elementId, tasks)` — element's tasks sorted by `sequenceOrder`
- `groupTasksByJob(tasks, elements)` — tasks grouped by Job

### Frontend Components

- `ElementSection.tsx` — element header in Job Details Panel (suffix, label, prerequisites)
- `TaskList.tsx` — tasks grouped by Element
- Single-element jobs: Element header hidden (`isSingleElement`)

### Backward Compatibility

- Single-element jobs (`isMultiElementJob() === false`) behave identically to pre-Element behavior
- Element layer is transparent for single-element jobs
- Currently (v0.4.0–v0.4.1) every Job has exactly one Element (1:1)
