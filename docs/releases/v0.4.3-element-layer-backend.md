## Overview

This release implements the **Element entity** in the PHP/Symfony backend, introducing the intermediate layer between Job and Task that enables multi-element jobs. Currently, the Element layer exists only in the TypeScript frontend (`@flux/types`, `@flux/schedule-validator`, `apps/web`). This release brings the PHP API into alignment by adding the Element entity, updating Job and Task relationships, creating database migrations, and updating services.

After this release, the PHP API will fully support the Job → Element → Task hierarchy that the frontend already implements.

## Technical Notes

### Current State (Pre-v0.4.3)

The PHP API has no Element-related code. The current structure is:

- **Job.php**: Aggregate root with `OneToMany` to Task, `requiredJobIds` for job-level dependencies
- **Task.php**: `ManyToOne` to Job (`job_id` column), `sequenceOrder` for ordering within Job
- **SnapshotBuilder.php**: Builds snapshots with `jobId` on tasks, no `elements` array
- **JobService.php**: Creates Job with tasks from DSL, no Element creation

### Target State (Post-v0.4.3)

- **Element.php** (NEW): Entity with `id`, `jobId`, `suffix`, `label`, `prerequisiteElementIds`, `createdAt`, `updatedAt`
- **Job.php**: `OneToMany` to Element (replaces direct Task relationship for traversal)
- **Task.php**: `ManyToOne` to Element (`element_id` column), Job accessible via `element.job`
- **SnapshotBuilder.php**: Includes `elements` array in snapshot, tasks reference `elementId`
- **JobService.php**: Auto-creates one default Element (suffix `"ELT"`) per Job

### TypeScript Reference (Source of Truth)

```typescript
// packages/types/src/element.ts
interface Element {
  id: string;
  jobId: string;
  suffix: string;           // e.g., "couv", "int", "fin", "ELT"
  label?: string;           // Optional display label
  prerequisiteElementIds: string[];
  taskIds: string[];        // Derived from tasks, not stored in PHP
  createdAt: string;
  updatedAt: string;
}
```

### Key Design Decisions

1. **Element fields in PHP**: `id` (GUID), `job` (ManyToOne to Job), `suffix` (string, max 20), `label` (nullable string, max 100), `prerequisiteElementIds` (JSON array), `createdAt`, `updatedAt`.

2. **`taskIds` is NOT stored on Element in PHP**: Unlike TypeScript where `taskIds` is a field, in PHP we use Doctrine's `OneToMany` relationship from Element to Task. The `taskIds` array is derived when building snapshots.

3. **Task.job removal is deferred**: Task keeps `element_id` FK, but we also keep `job_id` for backward compatibility during migration. The `getJobId()` method will navigate via `element.job.id`. The `job_id` column will be dropped in a later migration after all services are updated.

   **UPDATE**: Actually, to match the roadmap exactly ("Task.job removed, accessible via element.job"), we will:
   - Add `element_id` FK on Task
   - Remove `job_id` FK from Task
   - Update `Task.getJobId()` to navigate via `$this->element->getJob()->getId()`
   - Update Job entity: tasks accessed via `elements → tasks` (or keep a convenience method)

4. **Default Element suffix**: `"ELT"` (matches TypeScript convention for single-element jobs).

5. **sequenceOrder**: Stays 1-based in PHP (matching current Task entity), while TypeScript uses 0-based.

6. **Migration strategy**: Three-step migration in a single migration file:
   - Step 1: Create `elements` table
   - Step 2: For each existing Job, create one Element with suffix `"ELT"` and link its tasks
   - Step 3: Add `element_id` column to `tasks`, populate from data migration, then drop `job_id` column
