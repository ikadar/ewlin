## Overview

The precedence constraint lines (purple for earliest start, orange for latest start) currently do not account for non-working hours when calculating positions. This causes incorrect constraint visualization when lunch breaks, non-working hours, or weekends fall between the predecessor end and the calculated earliest/latest start time.

**Implements:** [REQ-03](../ux-ui/tmp/refactored-new-requirements-03-en.md#req-03-precedence-lines-should-respect-non-working-hours)

## Technical Notes

### Current Behavior (Incorrect)

```
Predecessor task ends: 11:30
Dry time: +4 hours
Current calculation: 15:30 ← WRONG (ignores lunch break 12:00-13:00)

Timeline:
11:00 ──┬── 12:00 ──┬── 13:00 ──┬── 14:00 ──┬── 15:00 ──┬── 16:00
        │ Pred ends │░░░░░░░░░░░│           │           │
        │   11:30   │  LUNCH    │           │  Purple   │
        │           │ (ignored) │           │  line at  │
        │           │           │           │  15:30    │
```

### Expected Behavior (Correct)

```
Predecessor task ends: 11:30
Dry time: +4 hours
Lunch break: 12:00-13:00 (1 hour non-working)
Correct calculation: 16:30 ← Accounts for lunch

Timeline:
11:00 ──┬── 12:00 ──┬── 13:00 ──┬── 14:00 ──┬── 15:00 ──┬── 16:00 ──┬── 17:00
        │ Pred ends │░░░░░░░░░░░│           │           │           │
        │   11:30   │  LUNCH    │           │           │  Purple   │
        │           │ (counted) │           │           │  line at  │
        │           │           │           │           │  16:30    │
```

### Data Structures

The station's `operatingSchedule` provides the working hours:

```typescript
interface OperatingSchedule {
  monday: DaySchedule;
  tuesday: DaySchedule;
  // ... etc
}

interface DaySchedule {
  isOperating: boolean;
  slots: TimeSlot[]; // e.g., [{start: "08:00", end: "12:00"}, {start: "13:00", end: "17:00"}]
}
```

### Algorithm: `addWorkingTime`

```typescript
/**
 * Add working time to a date, skipping non-working periods.
 *
 * @param startTime - Starting point
 * @param durationMs - Duration to add (in milliseconds)
 * @param station - Station with operatingSchedule and exceptions
 * @returns New Date after adding working time
 */
function addWorkingTime(
  startTime: Date,
  durationMs: number,
  station: Station
): Date {
  let current = new Date(startTime);
  let remainingMs = durationMs;

  while (remainingMs > 0) {
    const daySchedule = getDaySchedule(current, station);

    if (!daySchedule.isOperating) {
      // Skip to next day's start
      current = getNextWorkingDayStart(current, station);
      continue;
    }

    // Find current slot or next slot
    const slot = findCurrentOrNextSlot(current, daySchedule.slots);

    if (!slot) {
      // No more slots today, skip to next day
      current = getNextWorkingDayStart(current, station);
      continue;
    }

    // Calculate time available in this slot
    const slotEnd = parseTime(slot.end, current);
    const slotStart = parseTime(slot.start, current);

    if (current < slotStart) {
      // Jump to slot start
      current = slotStart;
    }

    const availableMs = slotEnd.getTime() - current.getTime();

    if (remainingMs <= availableMs) {
      // Fits in this slot
      current = new Date(current.getTime() + remainingMs);
      remainingMs = 0;
    } else {
      // Consume this slot and continue
      remainingMs -= availableMs;
      current = slotEnd;
    }
  }

  return current;
}
```

### Integration Points

1. **`getPredecessorConstraint`** in `precedenceConstraints.ts`:
   - After getting `earliestStart` from `getEffectivePredecessorEnd`
   - If the station has specific schedule, add working time consideration

2. **`getSuccessorConstraint`** in `precedenceConstraints.ts`:
   - When calculating `latestStart` backward from successor
   - Use `subtractWorkingTime` (reverse of addWorkingTime)

### Test Scenarios

#### Scenario 1: Lunch Break Handling

**Preconditions:**
- Task 1 (predecessor) ends at 11:30
- Task 1 is on a printing station (requires 4h dry time)
- Station has lunch break 12:00-13:00

**Steps:**
1. Load the fixture
2. Select the job
3. Start dragging Task 2
4. Observe the purple (earliest start) line position

**Expected Results:**
- [ ] Purple line appears at 16:30 (not 15:30)
- [ ] Calculation: 11:30 + 4h dry time = 15:30, but 12:00-13:00 is non-working
- [ ] So: 11:30 → 12:00 (30 min) + skip lunch + 13:00 → 16:30 (3h 30min) = 4h total

#### Scenario 2: End of Day Handling

**Preconditions:**
- Task ends at 15:30
- 4h dry time needed
- Station closes at 17:00, opens at 08:00 next day

**Steps:**
1. Load the fixture with task ending at 15:30
2. Start dragging successor task
3. Observe the purple line position

**Expected Results:**
- [ ] Purple line appears on the NEXT DAY at appropriate time
- [ ] Calculation: 15:30 → 17:00 (1.5h) + skip night + 08:00 → 10:30 (2.5h) = 4h

#### Scenario 3: Weekend Handling

**Preconditions:**
- Task ends on Friday at 16:00
- 4h dry time needed
- Station doesn't operate on weekends

**Steps:**
1. Load fixture with Friday task
2. Start dragging successor task
3. Observe the purple line position

**Expected Results:**
- [ ] Purple line appears on MONDAY
- [ ] Calculation: Friday 16:00 → 17:00 (1h) + skip weekend + Monday 08:00 → 11:00 (3h) = 4h
