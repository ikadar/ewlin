## Overview

The precedence constraint lines (purple for earliest start, orange for latest start) currently do not account for non-working hours when calculating positions. This causes incorrect constraint visualization when lunch breaks, non-working hours, or weekends fall between the predecessor end and the calculated earliest/latest start time.

**Implements:** [REQ-03](../ux-ui/tmp/refactored-new-requirements-03-en.md#req-03-precedence-lines-should-respect-non-working-hours)

## Technical Notes

### Key Insight: Drying is Physical, Work is Scheduled

**Drying time is a physical process** that continues regardless of whether the workshop is open.
The ink keeps drying during lunch, overnight, and on weekends.

**Work (task execution) requires human presence** and can only happen during working hours.

### Scenario 1: Drying ends during lunch break

```
Predecessor task ends: 8:30
Dry time: +4 hours (physical process)
Drying ends: 12:30 ← during lunch break (12:00-13:00)
Earliest work start: 13:00 ← snap to next working time

Timeline:
08:00 ──┬── 09:00 ──┬── 10:00 ──┬── 11:00 ──┬── 12:00 ──┬── 13:00
        │ Pred ends │           │           │░░░░░░░░░░░│ Purple
        │   8:30    │           │   Drying  │  LUNCH    │ line at
        │           │           │   ends    │  (can't   │ 13:00
        │           │           │   12:30   │   work)   │
```

### Scenario 2: Drying ends within working hours

```
Predecessor task ends: 11:00
Dry time: +4 hours (physical process)
Drying ends: 15:00 ← within working hours (13:00-22:00)
Earliest work start: 15:00 ← no snap needed

Timeline:
11:00 ──┬── 12:00 ──┬── 13:00 ──┬── 14:00 ──┬── 15:00
        │ Pred ends │░░░░░░░░░░░│           │ Purple
        │   11:00   │  LUNCH    │  Drying   │ line at
        │           │  (drying  │  ends at  │ 15:00
        │           │ continues)│  15:00    │
```

### Data Structures

The station's `operatingSchedule` provides the working hours:

```typescript
interface OperatingSchedule {
  monday: DaySchedule;
  tuesday: DaySchedule;
  // ... etc
}

interface DaySchedule {
  isOperating: boolean;
  slots: TimeSlot[]; // e.g., [{start: "08:00", end: "12:00"}, {start: "13:00", end: "17:00"}]
}
```

### Algorithm: Precedence Constraint Calculation

**For earliest start (purple line):**
```typescript
// 1. Drying is physical - simple addition
const dryingEnd = new Date(predecessorEnd.getTime() + DRY_TIME_MS);

// 2. Work can only start during working hours
// If drying ends outside working hours, snap to next working time
const earliestStart = snapToNextWorkingTime(dryingEnd, station);
```

**For latest start (orange line):**
```typescript
// 1. Drying is physical - simple subtraction from successor start
const latestEnd = new Date(successorStart.getTime() - DRY_TIME_MS);

// 2. Task execution uses working time (actual work)
const latestStart = subtractWorkingTime(latestEnd, taskDurationMs, station);
```

**Key functions:**
- `snapToNextWorkingTime(time, station)` - If time is outside working hours, snap to next available
- `subtractWorkingTime(time, duration, station)` - Subtract duration respecting working hours

### Integration Points

1. **`getPredecessorConstraint`** in `precedenceConstraints.ts`:
   - After getting `earliestStart` from `getEffectivePredecessorEnd`
   - If the station has specific schedule, add working time consideration

2. **`getSuccessorConstraint`** in `precedenceConstraints.ts`:
   - When calculating `latestStart` backward from successor
   - Use `subtractWorkingTime` (reverse of addWorkingTime)
