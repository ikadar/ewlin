## Overview

Extends the PHP API to expose outsourced task data via the existing
`GET /api/v1/flux/jobs` endpoint and adds a new
`PATCH /api/v1/flux/tasks/{taskId}/status` endpoint.

This is the backend half of the ST (Sous-traitance) column feature. The
frontend implementation follows in v0.5.23.

## Technical Notes

### Data model (no changes needed)

The `tasks` table already contains all required fields:

```
Task
├── id              GUID
├── element_id      FK → elements.id
├── provider_id     FK → outsourced_providers.id   (null for internal tasks)
├── action_type     VARCHAR(50)
├── task_type       'Internal' | 'Outsourced'
└── status          TaskStatus enum
```

`outsourced_providers.name` resolves the supplier display name.

### Status mapping

| Dashboard (`FluxSTStatus`) | Backend (`TaskStatus`) |
|---|---|
| `pending` | `defined`, `ready` |
| `progress` | `assigned` |
| `done` | `completed` |
| *(hidden)* | `cancelled` |

On **write** (PATCH), the mapping is:

| Dashboard → | Backend |
|---|---|
| `pending` → | `TaskStatus::defined` |
| `progress` → | `TaskStatus::assigned` |
| `done` → | `TaskStatus::completed` |

### Current blocker in `FluxElementResponse.php`

The DTO currently **skips** outsourced tasks:

```php
foreach ($element->getTasks() as $task) {
    if (!$task->isInternal()) {  // ← outsourced tasks filtered out
        continue;
    }
    // ... build station data
}
```

The fix: after the internal-tasks loop, add a separate outsourcing loop.

## Feature Checklist

### GET — Extend FluxElementResponse

- [x] **Add `outsourcing` array to `FluxElementResponse`**
  - Description: After the existing internal-tasks loop, iterate over outsourced tasks,
    resolve `providerName` via `$task->getProvider()->getName()`,
    map `TaskStatus` → dashboard `FluxSTStatus`, emit each as an entry.
    `cancelled` tasks are excluded. `outsourcing` is always present (empty `[]` if none).
  - File: `src/DTO/Flux/FluxElementResponse.php`
  - Tests: `tests/DTO/Flux/FluxElementResponseTest.php`

### PATCH — New endpoint

- [x] **New DTO `FluxTaskStatusRequest`**
  - Description: Validates that `status` ∈ `{pending, progress, done}`.
    Symfony validation constraint on `status` property.
  - File: `src/DTO/Flux/FluxTaskStatusRequest.php` *(new)*
  - Tests: `tests/DTO/Flux/FluxTaskStatusRequestTest.php` *(new)*

- [x] **Add `patchTaskStatus()` action to `FluxController`**
  - Description: Route `PATCH /api/v1/flux/tasks/{taskId}/status`.
    Load `Task` by `taskId`, return 404 if not found.
    Return 400 if `task_type` ≠ `Outsourced`.
    Map dashboard status → `TaskStatus` enum, persist via EntityManager.
    Return 204 No Content on success, 422 on invalid `status` value.
  - File: `src/Controller/Api/V1/FluxController.php`
  - Tests: `tests/Controller/Api/V1/FluxControllerPatchTaskStatusTest.php` *(new)*

### Tests

- [x] **Unit tests — FluxElementResponse outsourcing serialization**
  - Outsourced tasks appear in `outsourcing[]`
  - Cancelled outsourced tasks are excluded
  - Empty array when no outsourced tasks
  - Status mapping: `defined` → `pending`, `ready` → `pending`,
    `assigned` → `progress`, `completed` → `done`

- [x] **Unit tests — PATCH endpoint**
  - 204 on valid `pending` / `progress` / `done`
  - 400 when task is Internal (not Outsourced)
  - 404 when `taskId` does not exist
  - 422 when `status` is an invalid value

## v0.5.22 – ST Column: Backend API

### Summary

Exposes outsourced task data in the Flux Jobs API and adds a new endpoint

- None — `outsourcing` is a new additive field (empty array `[]` by default)
