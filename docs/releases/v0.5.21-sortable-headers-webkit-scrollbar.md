## Overview

Two spec compliance gaps identified in the Flux Dashboard (spec 3.6):

1. **Webkit scrollbar**: Only Firefox-specific CSS was present (`scrollbarWidth`/`scrollbarColor`). Chrome/Safari showed native scrollbars instead of the spec's custom thin overlay (6px, dark gray thumb).
2. **Column header sorting**: Chevrons were visible on hover but clicking did nothing (code comment: "non-functional in v0.5.15"). Spec requires column sort with direction toggle.

## Scope

### In Scope

- Webkit scrollbar CSS (`.flux-scrollable` class with pseudoelement rules)
- Sortable column headers: `id`, `client`, `designation`, `sortie`, `bat`, `papier`, `formes`, `plaques`, `transporteur`
- Sort state in FluxPage (not FluxTable) — consistent with tab filter + search state
- Active sort column shows blue up/down arrow (always visible); inactive sortable columns show double chevron on hover
- Default sort: `id` ascending (no breaking change)
- Prerequisite column sort uses `worstPrerequisiteStatus()` for multi-element jobs; ascending = best (green) first
- `sortie` date sort: DD/MM → MMDD key for correct lexicographic comparison
- `transporteur` null: sorts to end in ascending

### Out of Scope

- `sortie` full year context (current year assumption is acceptable)
- Parti column sorting (K5.1 deferred)
- Station column sorting (K9.2 deferred)
- URL persistence for sort state (later release)

## Technical Notes

- Sort state lives in `FluxPage` alongside tab filter and search state; `FluxTable` is a pure presentational component.
- `SortChevron` helper replaces the static `sortIcon` SVG for active/inactive visual state.
- The internal `sorted` useMemo in `FluxTable` is removed; `FluxPage` sorts via `sortFluxJobs()` before passing jobs down.
- `FluxTableContext` extended with `sortColumn`, `sortDirection`, `onSortChange` for header access.
- Webkit scrollbar applied via `.flux-scrollable` CSS class (not inline style), Firefox `scrollbarWidth`/`scrollbarColor` retained.
