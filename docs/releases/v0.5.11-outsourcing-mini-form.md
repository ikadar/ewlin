## Overview

This release adds an editable mini-form for outsourced tasks in the Job Details Panel. Since v0.5.10 removed provider columns from the grid, outsourced tasks are now managed exclusively through this form. The mini-form allows users to configure outsourcing parameters (work days, departure date, return date) directly within the TaskTile component.

This is a **UI-only release**. The form captures user input and stores it locally; API integration will follow in a later release.

## Technical Notes

### OutsourcedTask Type

The `@flux/types` package already has the required fields (from v0.5.9):

```typescript
interface OutsourcedTask extends BaseTask {
  type: 'Outsourced';
  providerId: string;
  actionType: string;
  duration: OutsourcedDuration;
  manualDeparture?: string;  // ISO datetime
  manualReturn?: string;     // ISO datetime
}

interface OutsourcedDuration {
  openDays: number;
  latestDepartureTime: string;  // HH:MM
  receptionTime: string;        // HH:MM
}
```

### OutsourcedProvider Type

Provider data (from v0.5.9):

```typescript
interface OutsourcedProvider {
  id: string;
  name: string;
  latestDepartureTime: string;  // HH:MM
  receptionTime: string;        // HH:MM
  transitDays: number;
}
```

### Mini-Form Design

The form is embedded directly in the TaskTile (not a modal or collapsible):

```
┌────────────────────────────────────────┐
│ ■ Pelliculage                          │  <- Header: action type
│   Provider: ABC Finitions              │  <- Provider name (read-only)
├────────────────────────────────────────┤
│   JO:  [3]      Work days input        │
│   Dép: [05/02 14:00] Departure picker  │
│   Ret: [12/02 09:00] Return picker     │
└────────────────────────────────────────┘
```

### Auto-Calculation Logic

When predecessor is scheduled, calculate departure/return:

1. **Departure**: If predecessor ends ≤ latestDepartureTime → same day; else → next business day
2. **Return**: departure + transitDays + workDays + transitDays (at receptionTime)

This calculation happens client-side for display. The actual scheduling constraint is handled by v0.5.12.

### State Management

For this release, use local component state. Changes trigger an `onChange` callback that the parent can use to update mock data or (later) dispatch API calls.
