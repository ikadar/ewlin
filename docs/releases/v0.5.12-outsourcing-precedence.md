## Overview

Integrate outsourced task timing into the precedence constraint system. When an outsourced task is a predecessor or successor, the constraint lines must account for transit days and work days, not just the scheduled start/end times.

This extends the precedence visualization (purple/orange constraint lines) to correctly handle outsourced tasks with their unique timing model (departure → transit → work → transit → return).

## Technical Notes

### Outsourced Task Timeline

```
predecessor end → [departure at latestDepartureTime]
                     ↓
                  [transit: T days]
                     ↓
                  [work: N days (openDays)]
                     ↓
                  [transit: T days]
                     ↓
              [return at receptionTime] → successor can start
```

### Forward Calculation (Predecessor is Outsourced)

When calculating `earliestStart(successor)`:
1. Get outsourced task's return time (from `manualReturn` or calculated)
2. Snap to next working time if successor is internal
3. Return Y position for the constraint line

### Backward Calculation (Successor is Outsourced)

When calculating `latestEnd(predecessor)`:
1. Get outsourced task's departure time (from `manualDeparture` or calculated)
2. Subtract predecessor duration (with working hours if internal)
3. Return Y position for the constraint line

### Manual Override Priority

If `manualDeparture` or `manualReturn` is set on the outsourced task, use those values instead of calculated values. This respects user's explicit date choices from the mini-form.
