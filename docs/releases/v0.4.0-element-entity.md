## Overview

This release introduces the Element layer in the **frontend only**: TypeScript types, mock data, unit tests, Playwright E2E tests, and test fixtures. The backend remains unchanged - this is a preparatory release to ensure all frontend code is ready for the Element layer before the backend migration.

**Why frontend first?** By updating types, mocks, and tests first, we can:
1. Validate the Element model design in isolation
2. Ensure all tests pass with the new structure
3. Minimize risk during backend migration (v0.4.2)

## Technical Notes

### Model Change

**Current (before):**
```
Job (1) ──────────────► (n) Task
     direct relationship
```

**New (after):**
```
Job (1) ──► (n) Element (1) ──► (n) Task
```

### TypeScript Types

```typescript
// packages/types/src/element.ts
export interface Element {
  id: string;
  jobId: string;
  taskIds: string[];
  createdAt: string;
  updatedAt: string;
}

// packages/types/src/task.ts - CHANGED
export interface BaseTask {
  id: string;
  elementId: string;  // Was: jobId
  sequenceOrder: number;
  // ... rest unchanged
}

// packages/types/src/assignment.ts - CHANGED
export interface ScheduleSnapshot {
  // ... existing fields
  elements: Element[];  // NEW (was empty array, now populated)
  // ...
}
```

### Mock Data Strategy

For v0.4.0, the mock generators will:
1. Create one Element per Job (1:1 for now)
2. Assign all Job's Tasks to that Element
3. Include elements array in snapshot

This mirrors what the backend migration (v0.4.2) will do.
